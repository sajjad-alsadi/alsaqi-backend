# Database-aligned enum validation and 404 handling for the correspondence API

> **Review scope note.** The working tree is clean ŌĆö `git status` reports "nothing to commit, working tree clean", both `git diff` and `git diff --staged` are empty, and `main` is in sync with `origin/main`. There are no uncommitted (staged or unstaged) changes and no base-branch divergence to diff against. This review therefore covers the latest committed unit of work, **HEAD = `f8f14d5`** ("feat(correspondence): harden API validation with database-aligned enums and 404 handling"), as the de-facto current change.

This change tightens the correspondence API in three ways: it replaces permissive `z.string()` validation with `z.enum()` backed by a new centralized constant module in `@alsaqi/shared`; it adds `NotFoundError` (404) detection to the update/delete service paths via affected-row counts; and it standardizes the in-process `incoming`/`outgoing` discriminator to lowercase, mapping to the capitalized values the database stores. Along the way it isolates n8n webhook dispatch behind try/catch, adds `deleted_at IS NULL` to several queries, rewrites the fragile string-substitution in the combined archive query, fixes a duplicate-admin-notification race in the login lockout path, and converts the CRUD orphan-file cleanup to `fs/promises`. A property-based schema-sync test and adjusted integration tests are included.

**Watch for:** the `POST /attachments` body field `correspondence_type` was flipped from lowercase to capitalized `['Incoming','Outgoing']` while every path param moved the *other* way (to lowercase), which breaks any client that previously sent lowercase here (confirmed); the `GET /archive` `type` query param now silently falls through to "combined" when given the pre-change capitalized value, with no validation (confirmed); the login lockout quietly stopped being a sliding window (likely, verify intended); and the new schema-sync test validates hand-copied schema duplicates rather than the real route schemas (confirmed).

## High-level view

The spine of this change is a casing convention: HTTP-facing identifiers for a correspondence record's kind are now lowercase (`incoming`/`outgoing`), validated at the route via a shared `typeIdParamSchema`, and the service maps them to the capitalized `Incoming`/`Outgoing` that the `correspondence_type` columns actually store. This is a sound direction, but the commit applies it asymmetrically ŌĆö the attachment **create** body now demands capitalized values, and the archive **list** `type` query has no validation and silently changes behavior on the old casing.

Enum validation is the headline feature and it is well-structured: one constant module, consumed by both the shared validators and the route schemas, with derived TypeScript types. The weak point is the test that is supposed to lock this down ŌĆö it reconstructs the route schemas as local copies instead of importing them, so it can't detect drift in the file it names.

The 404 work is real and consistent across update/delete, and `updateStatus` already threw on a missing record ŌĆö but `archive()` was edited in this same commit and left without either a 404 check or the `deleted_at` guard its siblings received, so it is now the odd one out.

Resilience improved in two independent spots: webhook dispatch failures no longer roll back committed writes, and the lockout path no longer notifies admins twice under concurrent failed logins. The lockout fix carries an unannounced side effect ŌĆö once locked, further failed attempts no longer extend the window.

Two items ride along that are unrelated to correspondence: a BullMQ readiness probe in `src/index.ts` and a Dockerfile base-image pin.

### Case-as-contract: the incoming/outgoing discriminator flipped to lowercase

The unifying idea of this change is that a correspondence record's kind travels as lowercase `incoming`/`outgoing` through the HTTP and service layers, and the service translates to the capitalized `Incoming`/`Outgoing` that the `correspondence_type` columns and the `'Incoming'`/`'Outgoing'` literals in queries expect. The new `typeIdParamSchema` enforces the lowercase set on every `:type/:id` route, and a `dbType = type === 'incoming' ? 'Incoming' : 'Outgoing'` mapping appears in `updateStatus`, `getAttachments`, and `getDetails`. The pre-change comparisons (`type === 'Incoming'`, `type === 'Outgoing'`) were flipped to lowercase throughout the service.

The diagram exposes the asymmetry. Path params standardized on lowercase, but `src/schemas/correspondence.ts` moved the attachment **body** field the opposite way ŌĆö `correspondence_type: z.enum(['Incoming','Outgoing'])`, changed from the prior `['incoming','outgoing']`. `addAttachment` inserts that value verbatim, so the database stays internally consistent with what `getAttachments`/`getDetails` read back. But the public contract for `POST /attachments` now rejects the lowercase value it accepted before this commit, and it disagrees with the lowercase `:type` used by the matching `GET /attachments/:type/:id`. A client that uploads with `correspondence_type: 'incoming'` will now get a 400 (confirmed). Either the body field should accept the same lowercase set as the path params and be mapped server-side, or the inconsistency should be called out and the frontend updated in lockstep.

The second exposure is `GET /archive`. The route hands `req.query` straight to `getArchive` with no `validateQuery`, and `getArchive` branches on `type === 'incoming'` / `type === 'outgoing'`, falling through to a combined incoming+outgoing query otherwise. Before this commit those comparisons were `'Incoming'`/`'Outgoing'`. So a caller still sending `?type=Incoming` ŌĆö the value that worked previously ŌĆö now silently receives the merged set instead of an incoming-only list, with no error to signal the mismatch (confirmed). The single-record routes are protected by `typeIdParamSchema`; this list route is not.

### 404 detection lands on update/delete but skips archive

`updateOutgoing`, `deleteOutgoing`, `updateIncoming`, and `deleteIncoming` now throw `NotFoundError` when their affected-row count is zero, and `updateStatus` reaches the same outcome through its existing `if (!oldRecord) throw new NotFoundError(...)`.

`archive()` is the gap. It was edited in this commit (the casing flip) but left as a bare `UPDATE ${table} SET is_archived = 1, status = 'Archived' ... WHERE id = ?::uuid` with no row-count check and no `deleted_at IS NULL` predicate. Archiving a non-existent id returns 200 with `{success:true}`, and archiving an already soft-deleted row will flip its archive flag and status back on a logically-deleted record (confirmed). Given the commit's own theme, `archive()` should get the same `result.changes === 0 ŌåÆ NotFoundError` treatment and the `deleted_at IS NULL` guard as the rows around it.

### Soft-delete filtering is mostly applied, with two holes

`deleted_at IS NULL` was added to the incoming list, the archive queries, `getOutgoing`'s count, and the update/delete `WHERE` clauses ŌĆö a consistent tightening. Two write paths edited in this commit were not brought along: `updateStatus` (both the `SELECT status` and the `UPDATE`) and `archive()` operate without the predicate, so a status transition or archive can still mutate a soft-deleted record (likely). If the soft-delete model is meant to make a record inert, these two are leaks in it.

### Dynamic partial updates replace full-row overwrites

`updateOutgoing` and `updateIncoming` were rewritten from fixed column lists to building `SET` clauses only for the fields present on the payload, driven by a hardcoded `fieldMap`. Column names come from that fixed map and values are bound parameters, so the dynamic SQL carries no injection surface. The behavioral change worth flagging: previously every column was written on every update (omitted fields became null), now omitted fields are preserved ŌĆö PATCH semantics. The `PUT /incoming/:id` route validates with `incomingSchema.partial()`, consistent with the new behavior, but any caller that relied on a PUT nulling out an omitted field will see different results (confirmed). Worth confirming this is the intended contract.

### Archive-list pagination bypasses the shared clamp

The other list endpoints (`/outgoing`, etc.) run their query through `parsePaginationParams`, but `GET /archive` passes raw `req.query` into `getArchive`. The rewritten combined branch then computes `effectivePageSize = Number(pageSize) || 15` and `effectivePage = Number(page) || 1` with no upper bound, so `?pageSize=100000000` becomes a literal `LIMIT 100000000` (likely ŌĆö an unbounded result-size / DoS lever). The single-type branches forward the raw `page`/`pageSize` (which arrive as query strings) into `qb.paginate(...)` and `computePaginationMeta(...)`; whether that misbehaves depends on coercion inside `QueryBuilder`, but it is at least inconsistent with the `Number(...)` handling in the sibling branch (possible). Routing `/archive` through the same `parsePaginationParams` clamp the other lists use would close both.

The same rewrite fixes a real bug worth crediting: the old combined query used `whereBlock.replace('search_column', ...)` string substitution and applied `qb.buildParams()` twice against a single placeholder set, which was fragile and easy to desync. Building two independent `QueryBuilder`s and concatenating their params in `UNION ALL` order removes the substitution hazard.

### Login lockout: notification dedup is solid, the window quietly stopped sliding

`handleFailedLogin` previously ran an unconditional `UPDATE users SET locked_until = ...` and then always called `notifyAdminsOfLockout`. Under concurrent failed logins past the threshold, that meant multiple requests each re-stamped the lock and each fired an admin notification. The new conditional `UPDATE ... WHERE id = ? AND (locked_until IS NULL OR locked_until < CURRENT_TIMESTAMP) RETURNING id` plus `if (lockResult)` ensures only the request that actually transitions the account into the locked state notifies ŌĆö a correct, security-relevant dedup.

The side effect is that the lockout is no longer a sliding window. Because the `WHERE` excludes rows already locked into the future, failed attempts that arrive *during* an active lock no longer push `locked_until` forward ŌĆö the lock now expires a fixed 15 minutes after the transition rather than 15 minutes after the most recent attempt (likely). For an already-locked account this is arguably fine (the attacker can't authenticate during the window regardless), but it is a behavioral change to an auth-security control that the commit message doesn't mention, so it should be a conscious decision. Separately and cosmetic: `notifyAdminsOfLockout` still composes its message from `user.failed_attempts + 1` (the pre-increment in-memory snapshot) rather than the authoritative post-increment count, so the reported attempt count can be slightly off.

### Webhook isolation logs to the wrong place

Every `N8nService.sendEvent(...)` in the service (create/update/delete for both types, plus `status_changed`) is now wrapped in try/catch so an automation failure can't roll back or fail an already-committed write ŌĆö the correct fix for finding 1.14 ŌåÆ 2.14. The catch blocks log via `console.error` (likely an inconsistency): the project ships a winston-based logger and an audit pipeline, so these dispatch failures bypass it and won't surface where operators look for backend errors.

### The schema-sync test doesn't exercise the real route schemas

The new `correspondence.schema-sync.test.ts` is a property-based check (fast-check, 100 runs/field) that every enum field accepts exactly its constant set and rejects everything else. The structural problem is in its setup: `routeIncomingSchema`, `routeOutgoingSchema`, and `routeLinkSchema` are **redeclared inside the test file** with a comment that they're "reconstructed exactly as in src/routes/correspondence.ts", then compared against the shared validators imported from `@alsaqi/shared`. Both sides import the same constant arrays, so the comparison largely confirms that `z.enum(CONST)` accepts `CONST` ŌĆö and it will keep passing even if someone edits the real schemas in `correspondence.ts`, because the test holds its own copy (confirmed). The commit message's claim that this verifies "route and service schemas remain aligned" overstates what it does. Importing the actual route schemas (exporting them from the route module if needed) would make the test guard the thing it names. The route-level `incomingStatusUpdateSchema`/`outgoingStatusUpdateSchema` and the attachment `correspondence_type` casing are not covered by any sync assertion.

### Test coverage: adjusted, not meaningfully expanded

The integration-test diff is mostly mechanical: ids changed from `inc-1`/`out-1` to valid UUIDs (now required by `typeIdParamSchema`'s integer-or-UUID refinement) and `new_status: 'In Progress'` changed to `'Under Review'` (since `'In Progress'` isn't in `INCOMING_STATUSES`). These keep the existing happy-path and missing-field cases green under the stricter validation, which is necessary, but the commit's "expand integration tests ŌĆ” with enum validation coverage" is thin in practice.

Not tested at the route level: rejection of an out-of-enum value (e.g. `priority: 'Bogus'`, `new_status` outside the type's allowed set); rejection of an invalid `:type` or malformed `:id`; the new 404 responses from update/delete/status; the archive-list `type` casing behavior; and the attachment-body capitalization. These are exactly the behaviors this commit introduces, so they're the ones most worth pinning down.

### Bundled, unrelated changes

`src/index.ts` gains a BullMQ readiness probe in `/ready` ŌĆö a dynamic import of `queueManager` reporting `ok`/`not_initialized`/`unavailable`, deliberately never flipping the overall `healthy` flag (queues are optional). The Dockerfile change pins `node:20-slim ŌåÆ node:20.19-slim`; pinning the minor helps, though without a digest the build still isn't byte-reproducible, so the "reproducible builds" comment slightly overstates it. Both are unrelated to correspondence hardening and would make a cleaner history split out from the API change.

### Patterns worth keeping

The enum centralization is the right shape: one `packages/shared/src/constants/correspondence.ts` as the single source of truth, re-exported through the shared barrel, consumed by both the shared validators and the route schemas, with derived `typeof X[number]` types so the TypeScript types can't drift from the runtime arrays. The commit also closed a real authorization gap ŌĆö `POST /link` previously ran on `authenticate` alone and now carries `checkPermission('Correspondence', 'Edit')`, matching the other mutating endpoints. And the `:id` refinement that accepts either an integer or a UUID is a sensible guard to add at the edge.
